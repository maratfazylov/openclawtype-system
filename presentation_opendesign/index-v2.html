Блок 2 · от LangChain к Deep Agents

Эволюция: от ответа к действию, затем к длинному циклу с проверкой.

Chatbot / RAG

Отвечает. Читает базу. Не действует во внешнем мире.

Tool-calling

Вызывает API. Может писать в БД. Но нет длинного цикла и самопроверки.

Deep Agent

Планирует и итерирует. Управляет файлами, shell, browser. Стопорится перед опасным действием.

Deep Agent = LLM + Tools + State + Loop + Human checkpoint
Заметки: Формулу проговорить как контракт. Если убрать state — будет tool call. Если убрать loop — будет разовый вызов. Если убрать HITL — будет небезопасная автоматизация.
Главный факт

Claude Code уже Deep Agent.

Если вы пользовались Claude Code, вы уже работали с агентом, который планирует, вызывает tools, меняет файлы, проверяет себя и просит подтверждение на опасных шагах.

Заметки: Связать с личным опытом аудитории. Разница воркшопа: мы залезаем внутрь harness и заменяем connector/runtime под себя.

Deep Agents Architecture

`create_deep_agent(model, tools, ...)` — это не функция-магия, а сборка слоев.

01

Model

LLM, routing, structured calls, stop conditions.

02

Tools

Typed adapters: files, shell, browser, APIs, connectors.

03

State

Messages, artifacts, memory, task context, durable traces.

04

Loop + HITL

Plan, act, observe, critique, checkpoint before risk.

LangChain
  → create_agent()
  → create_deep_agent()

Переход от «легко вызвать model + tools» к полному harness: sub-agents, backend, skills, memory, HITL и context management.

Заметки: Здесь важно не уходить в реализацию LangGraph слишком глубоко. Слайд нужен как карта слоев перед архитектурой клона.

Connector не меняет runtime

    """Send message via Telegram Bot API."""

Safety model

Что мы специально ограничиваем в клоне, чтобы он был учебным, а не опасным.

Dry-run first

Telegram send, DevOps commands и внешние API сначала возвращают payload, а не выполняют действие.

HITL checkpoint

Файлы, shell, network и credentials требуют явной границы: read-only, confirm, execute.

Local-first secrets

`.env` не попадает в чат, memory или logs. Connector не должен сериализовать ключи.

agent receives content
→ classifies risk
→ proposes tool call
→ waits on human checkpoint
→ executes or aborts

Именно эти ограничения превращают OpenClaw-идею из «магии в мессенджере» в управляемый инженерный runtime.

Заметки: Этот слайд связывает первую часть про инциденты с практикой. Без ограничений мы просто повторяем те же ошибки в меньшем масштабе.
